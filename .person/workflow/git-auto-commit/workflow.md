# Workflow: AI 기반 자동 Git 커밋 (git-auto-commit)

이 워크플로우는 AI가 프로젝트의 변경 사항을 분석하여 논리적으로 연관된 기능 단위로 그룹화하고, 표준 Conventional Commit 메시지를 생성하여 사용자의 승인 하에 자동으로 커밋을 수행하는 과정을 정의한다.

---

## 1. 인터뷰 결과 요약 (명세)

### 문제 정의
개발자가 직접 커밋 단위를 나누고 메시지를 작성하는 번거로움을 해결하기 위해, AI가 변경 사항을 분석하여 연관된 로직끼리 그룹화하고 표준 Conventional Commit 메시지를 생성한 뒤 사용자의 승인을 거쳐 커밋을 자동화하는 도구.

### 핵심 요구사항
1. **변경 사항 분석 및 그룹화**: 수정된 파일(Modified)과 새로 생성된 파일(Untracked)을 모두 포함하여 연관된 로직(예: Controller-Service-Repository)끼리 지능적으로 그룹화.
2. **Conventional Commit 메시지 자동 생성**: `feat:`, `fix:`, `refactor:` 등 표준 형식을 따르는 메시지 생성.
3. **사용자 검토 및 승인**: AI가 제안한 커밋 목록(단위 및 메시지)을 사용자에게 보여주고 승인을 받음.
4. **커밋 편집 기능**: 사용자가 제안된 메시지를 수정하거나 특정 그룹을 커밋에서 제외할 수 있는 기능.
5. **가변적 그룹화 로직**: 변경 사항이 많으면 기능 단위로 크게 묶고, 적으면 세밀하게 판단하여 최적의 커밋 단위 제안.

---

## 2. 실행 흐름 (Step-by-Step)

### Step 1: 변경 사항 수집
- `git status` 및 `git diff`를 통해 Modified, Added, Untracked 파일을 모두 수집한다.
- 파일의 경로와 변경된 코드 내용을 분석 대상으로 확보한다.

### Step 2: 논리적 그룹화 분석
- 수집된 파일들의 연관성을 분석한다.
    - 예: `BoardController.java`, `BoardService.java`, `Board.java`가 함께 수정되었다면 'Board 기능 수정'으로 그룹화.
- 변경 사항의 규모에 따라 그룹 크기를 조절한다.
    - 변경 사항이 많을 경우: 도메인/기능 단위로 크게 그룹화.
    - 변경 사항이 적을 경우: 구체적인 작업 단위로 세밀하게 그룹화.

### Step 3: 커밋 메시지 생성
- 각 그룹별로 Conventional Commit 규격(`type: description`)에 맞는 메시지를 생성한다.
    - `feat`: 새로운 기능 추가
    - `fix`: 버그 수정
    - `docs`: 문서 수정
    - `style`: 코드 포맷팅, 세미콜론 누락 등 (코드 변경 없음)
    - `refactor`: 코드 리팩토링
    - `test`: 테스트 코드 추가/수정
    - `chore`: 빌드 업무 수정, 패키지 매니저 설정 등

### Step 4: 사용자 승인 및 편집 인터페이스 제공
- 사용자에게 다음과 같은 형식으로 제안한다.
    ```text
    AI가 분석한 커밋 그룹입니다:

    [Group 1]
    - Files: src/.../BoardController.java, src/.../BoardService.java
    - Message: feat: 게시판 조회 기능 구현
    
    [Group 2]
    - Files: src/main/resources/application.properties
    - Message: chore: DB 접속 설정 변경

    위 내용으로 커밋하시겠습니까? (Y/n/edit)
    ```
- `edit` 선택 시 사용자가 메시지를 수정할 수 있도록 유도한다.

### Step 5: 커밋 실행
- 승인된 그룹별로 `git add`와 `git commit -m "[message]"`를 순차적으로 실행한다.
- 모든 커밋 완료 후 `git status`를 출력하여 최종 상태를 확인한다.

---

## 3. 프롬프트 전략 (AI 지침)

- **정교한 분석**: 단순히 파일 이름만 보지 말고, 코드 내의 클래스명이나 메서드 변경 사항을 통해 실제 연관성을 파악하라.
- **간결한 메시지**: 커밋 메시지의 제목은 50자 이내로, 요점을 명확히 전달하도록 생성하라.
- **안전 우선**: 모호한 변경 사항은 억지로 묶지 말고 별도의 커밋으로 분리하라.
